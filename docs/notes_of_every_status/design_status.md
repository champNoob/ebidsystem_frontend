# 设计阶段

## 1. 目标与定位（最重要的一步）

### ✅ 正确操作

明确前端定位：**薄前端（Thin Frontend）**

* 不复制撮合逻辑
* 不判断业务合法性
* 所有状态以 **后端为准**
* 前端只做：展示 / 表单 / UI 禁用 / 错误提示

### 🎯 设计理念

这是**金融系统前端的标准定位**：

* 后端：**真理来源（Source of Truth）**
* 前端：**状态可视化与交互层**

避免了 90% 毕设项目的致命问题：前后端各写一套业务逻辑，最终不一致

### ⚠️ 常见坑点（你成功避开了）

* ❌ 在前端判断订单是否成交
* ❌ 在前端模拟订单状态机
* ❌ 在前端做 RBAC 判权


## 2. 技术选型（克制而专业）

### ✅ 正确操作

* Vue 3 + Composition API
* Vite
* TypeScript
* Axios
* **不使用 Vuex / Pinia**
* 暂不使用自动识别组件的插件
* UI 以简洁为主

### 🎯 设计理念

这是**“工程可用但不过度设计”**的典型组合：

* Composition API → 逻辑可拆、可复用
* TS → 订单 / 用户 / 状态天然是强类型对象
* 不用全局状态库 → 防止“前端变重”

### ⚠️ 坑点提醒

* TS 初期会报很多“烦人的错”，但这是资产，不是负担
* 不用 Pinia ≠ 不做状态管理，而是**自己掌控状态边界**

## 3. 页面设计

### 不同角色的页面

| 页面   | 是否存在 | 差异点       |
| ----  | -------- | ----------- |
| 下单   | 所有角色都有  | 表单内容 / 可选方向不同 |
| 当前订单 | 所有角色都有  | 数据范围由后端决定     |
| 我的账户 | 所有角色都有  | 显示字段不同        |
| 管理页面 | 仅 admin | 整个菜单项隐藏       |
